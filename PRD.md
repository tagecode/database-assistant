# BIU Database 产品需求文档（PRD）

## 1. 文档信息

- 产品名称：`BIU Database`
- 产品定位：跨平台桌面数据库管理客户端
- 文档版本：`v1.0`
- 文档状态：初稿
- 目标平台：`Windows / macOS / Linux`
- 技术基线：`Electron + React + TypeScript + MUI`

## 2. 产品概述

### 2.1 产品背景

开发者、测试、数据分析师和运维工程师在日常工作中需要频繁连接多种数据库完成查询、数据查看、结构维护、导入导出和排障等操作。现有数据库客户端虽然功能成熟，但通常存在以下问题：

- 产品价格较高，授权模式对个人和中小团队不够友好。
- 对国产化、自定义工作流和团队场景的支持不够灵活。
- 多数据库统一体验存在差异，特定场景扩展成本高。
- UI 交互与现代桌面工具体验相比存在优化空间。

因此，需要打造一款类 Navicat 的现代化桌面数据库管理客户端，在保证专业能力的同时提供更好的跨平台体验、可扩展性和后续商业化基础。

### 2.2 产品定位

`BIU Database` 是一款面向开发者和数据相关岗位的跨平台数据库客户端，提供数据库连接管理、SQL 编辑执行、对象树浏览、数据表查看与编辑、结果集分析以及导入导出等能力，目标是成为一款“开箱可用、体验现代、可持续扩展”的桌面数据库工作台。

### 2.3 产品愿景

打造一个兼具以下特性的数据库客户端：

- 上手快：第一次使用即可完成连接、查询和数据查看。
- 足够专业：满足日常开发、测试、排障和轻量管理需求。
- 多数据库统一：在不同数据库之间提供一致的核心操作体验。
- 易扩展：支持未来增加数据库类型、插件能力和企业级特性。

## 3. 产品目标

### 3.1 业务目标

- 提供一款可覆盖主流开发数据库场景的跨平台桌面客户端。
- 在 MVP 阶段形成可用闭环，支持连接、查询、浏览、编辑、导出。
- 为后续企业版功能打下架构基础，包括团队协作、权限控制、审计、插件化等。

### 3.2 用户目标

用户能够在一个客户端中完成以下核心工作：

- 管理多个数据库连接。
- 编写并执行 SQL。
- 浏览数据库、Schema、表、视图等对象。
- 查询并编辑表数据。
- 导入和导出常见数据格式。

### 3.3 MVP 成功标准

- 用户可在 3 分钟内完成一个数据库连接并执行首条 SQL。
- 常见查询场景下具备稳定响应能力。
- 多标签页工作流清晰，基本操作无需学习成本。
- 首版支持的数据库类型能覆盖大多数开发与测试场景。

## 4. 用户与使用场景

### 4.1 目标用户

1. 开发工程师  
需要频繁连接本地、测试和生产受控环境数据库，进行建表、查数、排障和 SQL 调试。

2. 测试工程师  
需要快速查看和修改测试数据，验证业务流程，执行辅助脚本。

3. 数据分析师 / 数据运营  
需要执行查询、导出结果、查看表结构和做轻量数据验证。

4. DBA / 运维工程师  
需要进行连接管理、结构查看、执行脚本、导出数据和问题定位。

### 4.2 典型场景

1. 用户创建数据库连接，保存连接配置，并快速连接常用环境。
2. 用户在 SQL 编辑器中编写查询语句并执行，查看结果集。
3. 用户通过左侧对象树浏览数据库、Schema、表、视图和字段。
4. 用户打开某张表的数据页，分页浏览并进行单元格编辑。
5. 用户将查询结果导出为 CSV/Excel/SQL 文件。
6. 用户查看建表语句、索引、字段类型等结构信息。

MVP 核心功能包括：

### 5.2 后续版本扩展范围

## 6. 核心需求清单

### 6.1 连接管理

#### 6.1.1 功能描述

用户能够创建、编辑、删除、测试并分组管理数据库连接。

#### 6.1.2 功能要求

- 支持按数据库类型创建连接。
- 支持基础连接参数配置：
  - 名称
  - 主机
  - 端口
  - 用户名
  - 密码
  - 默认数据库
  - 字符集（按数据库类型决定是否展示）
  - SQLite 文件路径
- 支持“测试连接”。
- 支持连接收藏、分组和搜索。
- 支持最近连接记录。
- 支持连接失败时给出明确错误提示。

#### 6.1.3 安全要求

- 密码不得明文持久化到普通配置文件。
- 敏感信息应通过系统安全存储方案保存。
- 渲染层不得直接操作数据库驱动。

### 6.2 对象树浏览

#### 6.2.1 功能描述

用户连接数据库后，可在左侧对象树中浏览数据库对象。

#### 6.2.2 功能要求

- 支持按连接展示数据库或实例。
- 支持展开 Schema / Database / Table / View / Index 等层级。
- 支持搜索对象名称。
- 支持右键菜单操作：
  - 新建查询
  - 刷新
  - 查看结构
  - 查看数据
  - 复制名称
- 支持懒加载，避免首次加载过慢。

### 6.3 SQL 编辑器

#### 6.3.1 功能描述

用户可在工作区打开一个或多个 SQL 编辑标签页，进行脚本编写、执行和结果查看。

#### 6.3.2 功能要求

- 基于 `Monaco Editor` 实现。
- 支持语法高亮。
- 支持行号、代码折叠、快捷键、自动缩进。
- 支持执行全部 SQL。
- 支持执行选中 SQL。
- 支持中止执行中的查询。
- 支持多个编辑标签页并行存在。
- 支持基础 SQL 格式化能力。
- 支持编辑器内容草稿自动保存。

#### 6.3.3 未来增强

- 智能补全
- 基于元数据的提示
- 错误定位
- 多数据库方言支持增强

### 6.4 查询执行与结果展示

#### 6.4.1 功能描述

用户执行 SQL 后，系统应返回执行结果并在结果面板展示。

#### 6.4.2 功能要求

- 支持展示查询成功/失败状态。
- 支持展示执行耗时、影响行数、返回行数。
- 支持多个结果集展示。
- 支持空结果提示。
- 支持错误信息展示，错误信息应尽量保留原始数据库错误上下文。
- 对大结果集需支持分页、虚拟滚动或分段加载。
- 结果区支持复制单元格、复制行、导出结果。

### 6.5 表数据浏览与编辑

#### 6.5.1 功能描述

用户可直接打开数据表并浏览、筛选、分页和编辑数据。

#### 6.5.2 功能要求

- 支持按表打开“数据”标签页。
- 支持分页查询。
- 支持排序、列宽调整、列显隐。
- 支持单元格编辑。
- 支持新增行、删除行。
- 支持提交前确认。
- 支持主键/唯一键识别，以便生成更新条件。
- 编辑失败时需提示失败原因并保留现场。

#### 6.5.3 实现建议

- 数据表格组件选型采用 `AG Grid`。
- 对大数据量必须启用虚拟滚动能力。

### 6.6 表结构查看

#### 6.6.1 功能描述

用户可查看表字段、索引、主键、注释和建表语句等元信息。

#### 6.6.2 功能要求

- 支持字段列表查看。
- 支持字段类型、是否为空、默认值、注释展示。
- 支持索引与主键信息展示。
- 支持查看建表语句。
- 支持复制 DDL。

### 6.7 导入导出

#### 6.7.1 导出要求

- 支持导出查询结果。
- 支持导出格式：
  - CSV
  - Excel（后续增强）
  - SQL Insert（后续增强）
  - JSON（可选）
- 支持选择导出列。
- 支持导出进度提示。

#### 6.7.2 导入要求

- 支持 CSV 导入到指定表。
- 支持字段映射配置。
- 支持预览导入内容。
- 支持导入失败明细反馈。

### 6.8 工作区与多标签页

#### 6.8.1 功能描述

应用采用多标签页工作区，支持同时处理多个连接和任务。

#### 6.8.2 功能要求

- 支持 SQL 标签页、表结构标签页、数据标签页。
- 支持关闭当前、关闭其他、关闭全部。
- 支持标签页未保存状态提示。
- 支持标签页切换记忆。

### 6.9 设置中心

#### 6.9.1 功能要求

- 支持浅色/深色主题切换。
- 支持编辑器字体大小配置。
- 支持表格每页默认条数配置。
- 支持查询超时默认值配置。
- 支持语言设置预留。

## 7. 信息架构与界面结构

### 7.1 应用整体布局

建议采用典型数据库客户端工作台布局：

- 左侧：连接与对象树面板
- 中间：主工作区（多标签页）
- 底部：结果面板 / 日志 / 执行信息
- 顶部：工具栏与全局操作入口

### 7.2 一级导航模块

1. 连接管理
2. SQL 工作台
3. 数据浏览
4. 结构查看
5. 导入导出
6. 设置中心

### 7.3 设计原则

- 优先服务高频操作，减少点击层级。
- 保持桌面工具风格，强调信息密度与可读性平衡。
- 所有长任务都应有明确状态反馈。
- 对破坏性操作必须二次确认。

## 8. 技术方案要求

### 8.1 技术栈选型

- 桌面框架：`Electron`
- 前端框架：`React + TypeScript`
- UI 组件库：`MUI`
- 状态管理：`Zustand`
- SQL 编辑器：`Monaco Editor`
- 数据表格：`AG Grid`
- 构建工具：`Vite`

### 8.2 数据库驱动层

Node 侧数据库驱动规划如下：

- MySQL：`mysql2`
- PostgreSQL：`pg`
- SQLite：`better-sqlite3`
- MongoDB：`mongodb`（后续）
- SQL Server：`mssql`（后续）

### 8.3 架构约束

- 数据库驱动仅运行于 Electron 主进程或受控 worker。
- 渲染进程通过 `preload + IPC` 访问能力。
- 不允许渲染进程直接持有数据库连接实例。
- 数据库能力需采用适配器模式封装，不可与 UI 直接耦合。

### 8.4 推荐模块拆分

- `renderer`：UI、页面、状态、交互
- `main`：窗口管理、IPC 注册、任务调度、系统能力
- `preload`：安全能力桥接
- `db adapters`：不同数据库的连接、元数据、查询能力封装
- `shared`：跨端共享类型定义

## 9. 非功能需求

### 9.1 性能

- 常见对象树首次加载需在可接受时间内完成。
- 查询结果展示需支持大结果集场景，避免 UI 卡死。
- 查询执行不得阻塞主 UI 交互。
- 大数据量渲染必须依赖虚拟滚动。

### 9.2 稳定性

- 查询失败、连接失败、驱动报错需可恢复。
- 应用崩溃后应尽可能恢复最近工作区状态。
- 长任务需支持取消或中断。

### 9.3 安全

- 用户凭据需安全存储。
- 日志中不得直接打印明文密码。
- 与数据库相关的高危操作应保留明确确认提示。
- IPC 接口需遵循最小暴露原则。

### 9.4 可维护性

- 核心模块具备明确边界。
- 数据库适配层支持按数据库类型独立扩展。
- 关键流程应具备基础日志与错误定位能力。

### 9.5 可扩展性

- 后续新增数据库类型时，不应大规模改动 UI 主流程。
- UI 层应尽量基于统一领域模型消费数据。

## 12. 验收标准

### 12.1 功能验收

- 用户可创建并保存连接。
- 用户可成功连接 MySQL / PostgreSQL / SQLite。
- 用户可在 SQL 编辑器中执行语句并查看结果。
- 用户可查看表结构和表数据。
- 用户可对数据进行基础编辑并提交。
- 用户可导出查询结果。

### 12.2 体验验收

- 主要操作路径清晰，无明显理解门槛。
- 关键状态有可感知反馈。
- 错误提示可帮助定位问题。
- 多标签页场景下工作流连续自然。

### 12.3 技术验收

- Windows / macOS / Linux 可打包运行。
- 架构符合主进程 / preload / 渲染层隔离原则。
- 敏感信息安全存储。
- 大结果集场景下界面无明显卡死。

## 13. 风险与应对

### 13.1 多数据库抽象复杂

风险：不同数据库在元数据、方言、分页和错误处理上差异显著。  
应对：先聚焦 MySQL / PostgreSQL / SQLite，建立统一适配器接口。

### 13.2 大结果集性能风险

风险：大量数据渲染导致内存与界面卡顿。  
应对：分页、虚拟滚动、分段加载、限制默认查询量。

### 13.3 原生模块打包风险

风险：`better-sqlite3` 等原生模块在跨平台打包中存在兼容成本。  
应对：尽早验证打包链路，并纳入持续构建流程。

### 13.4 安全风险

风险：密码存储、误操作执行、IPC 暴露过宽。  
应对：安全存储、二次确认、最小权限接口设计。

## 14. 里程碑建议

1. 完成项目基础架构与主进程通信能力。
2. 完成连接管理与连接测试。
3. 完成对象树与元数据读取。
4. 完成 SQL 编辑执行与结果展示。
5. 完成表数据浏览与基础编辑。
6. 完成导入导出与设置中心。
7. 完成跨平台打包与首轮稳定性验证。

## 15. 附录：MVP 推荐实现原则

- 先做高频核心能力，不做“看起来高级但低频”的功能。
- 优先保证稳定、清晰、可扩展，而不是首版一次性做全。
- 所有数据库功能都应经过统一领域抽象后再提供给 UI。
- 所有长耗时任务都应异步化，并提供进度与取消能力。
